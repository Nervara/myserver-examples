# Backup & Restore Walkthrough (through myserver)

This walkthrough was executed end-to-end against a live myserver instance
(`app.serverops.cloud`, team 10, server `green`). It deploys this sample as a
real app, takes a **real backup through the product**, adds more data, restores,
and shows the row count and `max(created_at)` revert to the exact backup point.

It answers the customer's question directly: **"where are the actual backups?"**
— a backup *Run* produces a concrete backup **execution** artifact (timestamp,
size, status), and *Restore* replays one of those executions.

| Resource | Detail |

| URL | `https://tweetgen.72.61.203.163.sslip.io` (live count + latest-10 page) |

The app writes one tweet (generated by DeepSeek, canned fallback on failure)
into `tweets` every ~5 seconds.

## How myserver backs up Postgres

A myserver Postgres backup is a **logical dump** of the database container,
identical in mechanism to:

```
# backup  (encrypted variant adds: | openssl enc -aes-256-cbc -salt -pbkdf2 -pass pass:<key>)
docker exec <db-container> pg_dump -Fc -U <user> <db> | gzip > <artifact>

# restore
gunzip -c <artifact> | docker exec -i <db-container> pg_restore -U <user> -d <db> --clean --if-exists
```

`pg_restore --clean --if-exists` drops and recreates the objects, so the
database is returned to *exactly* the state captured in the dump.

## The actual backup execution

Running the schedule once (Backups → Run) produced a real execution — this is
the artifact the customer should be able to see and restore from:

```
execution id : 1922
status       : completed
size         : 3424 bytes
filename     : tweetdb-postgresql-20260529-093855.sql.gz.enc   (gzip + AES-256-CBC)
message      : Backup completed successfully
```

## The three counts (proof the restore reverted the data)

The generator was paused (app *stop*) only at the measurement boundaries so the
counts are exact; `pg_dump` itself is a consistent snapshot.

| Stage | `count(*)` | `max(created_at)` |
| --- | --- | --- |
| **BACKUP POINT** (right before Run) | **54** | `2026-05-29 09:38:37.56519+00` |
| **AFTER MORE WRITES** (let it run, then stop) | **65** | `2026-05-29 09:40:30.265069+00` |
| **AFTER RESTORE** (from execution 1922) | **54** | `2026-05-29 09:38:37.56519+00` |

Both the count **and** `max(created_at)` reverted to the exact backup point
(65 → 54). After restore the app was started again and resumed inserting on top
of the restored rows (54 → 56 → …), confirming normal operation continues.

## Reproduce it (through the product)

All steps go through the myserver API / CLI — no direct container access.

1. **Deploy** this sample (see `README.md`): create the Postgres DB, create the
   app (`build_pack=dockerfile`, `base_directory=/samples/tweet-generator`,
   `ports_exposes=8080`), set the env vars, deploy. Open the URL — tweets flow.
2. **Create a backup schedule** for the database (Backups page), pointing at an
   S3 storage.
3. **Read the count** (myserver DB query, or the app's home page):
   `SELECT count(*), max(created_at) FROM tweets;` → record the **backup point**.
4. **Run** the backup → a backup *execution* appears (status `completed`, with a
   timestamp + size). This is "the actual backup".
5. Let the app write more tweets; read the count again → it is higher.
6. **Restore** from the execution recorded in step 4.
7. Read the count again → it reverts to the backup point. The app keeps running
   and continues inserting afterward.

## Notes on the API call

DeepSeek-V4-Pro is a reasoning model. On the Anthropic-shaped Messages endpoint
(`/v1/messages`) it must be called with `thinking: {"type": "disabled"}`,
otherwise it spends the whole token budget on a `thinking` block and returns no
tweet text. `app.py` sets this; without it every tweet falls back to canned text.
